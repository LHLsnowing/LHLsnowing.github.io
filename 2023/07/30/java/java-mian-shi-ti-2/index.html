一、Spring设计模式

  • 依赖注入（Dependency Injection）：依赖注入是Spring框架的核心概念和设计模式。它通过在对象之间注入依赖关系，而不是在类内部创建这些依赖关系的实例，实现了松耦合和可扩展性。Spring使用依赖注入来管理对象之间的关系，通过配置文件或注解来声明依赖关系。
  • 单例模式（Singleton Pattern）：Spring容器中管理的Bean默认是单例的，即每个Bean定义只会有一个实例。这样做可以减少对象的创建和销毁成本，并确保在整个应用程序中共享同一个实例。
  • 工厂模式（Factory Pattern）：Spring框架广泛使用工厂模式来创建和管理Bean对象。BeanFactory和ApplicationContext作为工厂类，负责创建和管理Bean实例。
  • 代理模式（Proxy Pattern）：Spring AOP（面向切面编程）基于代理模式实现。它通过动态代理技术，为目标对象创建一个代理对象，可以在目标对象的方法执行前后添加额外的逻辑，如事务管理、日志记录等。
  • 观察者模式（Observer Pattern）：Spring框架中的事件机制基于观察者模式。通过定义事件源（Publisher）和监听器（Subscriber），实现了一种松耦合的事件通知机制。

二、RDB 和 Master

1.RDB

RDB（Redis Database）是Redis的一种持久化方式，它将内存中的数据以二进制形式保存到磁盘上，以便在Redis重启后可以重新加载数据。RDB是Redis的默认持久化方式，相比于AOF（Append-Only File），RDB更加紧凑和高效。

RDB持久化的过程如下：

  • Redis服务器根据设定的策略（如指定间隔时间、修改次数等）自动触发持久化操作，或者通过手动执行SAVE或BGSAVE命令来进行持久化。
  • 在执行持久化操作时，Redis会fork出一个子进程，用于处理持久化过程，同时父进程继续处理客户端请求。
  • 子进程将当前内存中的数据以快照的形式写入到一个临时RDB文件中。
  • 当子进程完成数据写入后，会使用原子性操作将临时文件替换掉旧的RDB文件，确保数据的完整性。
  • 完成持久化后，Redis服务器继续处理客户端请求。

RDB持久化的优点包括：

  • 性能好：RDB是将数据以二进制快照的形式保存到磁盘上，读写速度较快，适合用于备份和恢复大规模数据。
  • 文件紧凑：RDB文件是紧凑且压缩的，相对于AOF文件会占用较少的存储空间。
  • 适合灾难恢复：RDB可以定期备份数据，并在需要时快速恢复，适合灾难恢复和数据迁移操作。
    RDB方式也有一些缺点：
  • 数据丢失：如果Redis发生突然崩溃，最后一次持久化之后的数据将会丢失。
  • 不适合频繁的写操作：由于RDB是定期持久化的方式，如果频繁进行写操作，可能会导致较长时间内的数据丢失。
  • 恢复速度较慢：RDB文件需要加载到内存中才能进行读取，如果数据量较大，可能会导致较长的恢复时间。

总的来说，RDB是Redis的一种持久化方式，适合用于备份、灾难恢复和数据迁移等操作，但需要注意数据丢失和恢复速度的问题。在实际应用中，可以根据具体场景选择合适的持久化方式（如结合使用RDB和AOF）以满足需求。

2.Master

在Redis中，Master（主节点）是指在主从复制（Master-Slave Replication）架构中负责处理写操作，并将数据同步到从节点的节点。主节点接收客户端的写请求，并将写操作记录到AOF日志或RDB文件中，然后将写命令发送给连接的从节点进行数据同步。

主节点具有以下主要功能和特点：

  • 数据写入：主节点负责接收客户端的写操作，并将写命令记录到持久化日志（如AOF日志或RDB文件）中，确保数据持久化和恢复能力。
  • 数据同步：主节点将写命令发送给连接的从节点，进行数据同步。从节点会执行相同的写操作，从而达到数据在主从节点之间的一致性。
  • 故障切换：当主节点发生故障或不可用时，Redis可以进行自动或手动的故障切换。Redis Sentinel或Redis Cluster能够检测到主节点的故障，并从从节点中选出新的主节点继续处理写操作。
  • 高可靠性：通过主从复制，主节点可以提供高可靠性。即使主节点发生故障，从节点可以被晋升为新的主节点，继续提供服务并保持数据一致性。

作为主节点，它对应的从节点是其数据的副本，一般不会直接处理客户端的读请求。客户端可以通过连接到主节点或从节点来执行读操作，但在写入数据时，只能通过主节点来保证数据的一致性。

总结起来，Redis中的主节点负责接收写操作并进行数据同步，具有高可靠性和数据一致性的特点。它是Redis主从复制架构中的核心组件之一，用于提供数据持久化、故障切换和数据同步的功能。

三、哈希环

哈希环（Hash Ring）是一种用于分布式系统中数据分片和负载均衡的数据结构。它通过将节点和数据映射到一个固定大小的环上来实现。
在哈希环中，每个节点和数据都被映射到环上的一个位置。节点可以是物理服务器、虚拟节点或其他资源，而数据可以是键值对、对象等。映射时使用哈希函数将节点和数据的标识映射到环上的某个点。

哈希环的主要应用场景之一是用于构建分布式缓存系统，例如一致性哈希算法（Consistent Hashing）。在一致性哈希算法中，节点和数据的映射关系能够提供以下好处：

  • 数据分片：哈希环将数据均匀地分布在各个节点上，使得每个节点负责处理一部分数据，实现了数据的分片存储。
  • 负载均衡：当有新节点加入或节点离开时，由于哈希函数的平滑性质，只有少量的数据需要重新映射，大部分数据保持不变。这样可以避免数据迁移的成本，实现了负载均衡。
  • 容错性：由于数据在哈希环上的分布较为均匀且节点数量较多，当某个节点出现故障或离线时，只需将其对应的数据映射到相邻的节点，不会影响整个系统的稳定性。

在哈希环中节点和数据的映射并不是完全随机的，而是按照哈希函数的规则进行分配。这样可以保证同一个节点或数据在多次映射中都能得到相同的结果，从而保证数据的一致性。

总的来说，哈希环是一种用于实现数据分片和负载均衡的数据结构，在构建分布式系统、缓存系统等场景中具有重要的作用。它通过哈希函数和环形结构的设计，提供了高效、可扩展和容错的数据分布方案。

2.双重锁的单例设计模式

双重锁（Double-Checked Locking）是一种在多线程环境下实现单例设计模式的优化方式。其核心思想是通过在加锁前后进行双重检查，以减少锁竞争的开销。

GPT写的实例

public class Singleton {
    private volatile static Singleton instance;

    private Singleton() {
        // 私有构造函数，防止外部实例化
    }

    public static Singleton getInstance() {
        if (instance == null) {  // 第一次检查，避免不必要的同步
            synchronized (Singleton.class) {
                if (instance == null) {  // 第二次检查，确保只创建一个实例
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
  1. 将实例变量 instance 声明为 volatile 类型，保证可见性，防止指令重排带来的问题。
  2. 第一次检查 instance == null 是为了避免不必要的同步，如果实例已经存在，则直接返回。
  3. 在同步块内进行第二次检查 instance == null，以确保只有一个线程创建实例。
  4. 使用了 synchronized 关键字来对对象进行加锁，保证线程安全。

3.SpringCloud你了解哪些？

Spring Cloud是一个基于Spring框架的开源微服务架构工具集，它为开发人员提供了一组用于构建分布式系统和微服务架构的解决方案。下面是一些常见的Spring Cloud组件和功能：

  • Eureka：服务注册和发现组件，实现了服务的自动注册和发现。
  • Ribbon：客户端负载均衡组件，可以根据配置策略选择合适的服务实例进行调用。
  • Feign：声明式、模板化的HTTP客户端，简化了服务之间的通信。
  • Hystrix：容错和熔断器组件，提供了线程池隔离、服务降级等功能，处理分布式系统中的故障和延迟问题。
  • Zuul：API网关组件，提供了路由、过滤、负载均衡等功能，用于对外暴露统一的接口。
  • Config：分布式配置管理组件，可以集中管理应用程序的配置文件，并支持动态刷新配置。
  • Bus：消息总线组件，可以实现配置的动态刷新和通知，减少配置更新的复杂性。
  • Sleuth：分布式请求链路跟踪组件，可用于追踪请求在微服务架构中的流转情况。
  • Zipkin：分布式链路追踪系统，可以收集、存储和展示分布式系统中的请求追踪信息。

4.Mysql索引失效的原因？

  1. 不合适的索引设计
  2. 使用函数或表达式进行查询：当在查询条件中使用函数或表达式时，MySQL无法直接使用索引进行快速查找，而会对每一行进行计算来判断是否满足条件，从而导致索引失效。
  3. 数据量过小：当表中的数据量非常小的时候，即使有索引存在，MySQL也可能选择全表扫描而不是使用索引，这是因为使用索引需要进行额外的IO操作，而全表扫描则可以更快地获取到数据。
  4. 索引列数据类型不匹配：索引列的数据类型与查询条件中的数据类型不匹配，例如将字符串与数字进行比较，MySQL可能无法使用索引。
  5. 隐式数据类型转换:查询条件中的数据类型与索引列的数据类型不匹配
  6. 数据分布不均匀：如果索引列的数据分布不均匀，可能会导致一部分索引无法有效地过滤掉不满足条件的行，从而导致索引失效。

5.WebSocket的使用?

WebSocket是一种在客户端和服务器之间进行全双工通信的协议，它使用单个TCP连接来实现双向通信。相比传统的HTTP请求-响应模式，WebSocket具有更低的延迟和更高的效率。

下面是使用WebSocket的基本步骤：

客户端与服务器建立WebSocket连接：

  • 在客户端，使用JavaScript代码创建一个WebSocket对象，并指定连接的URL。
  • 在服务器端，实现WebSocket服务器，接受客户端的连接请求。

WebSocket握手过程：

  • 客户端发送一个特殊的HTTP请求，请求协议头中包含Upgrade字段，指定升级为WebSocket协议。
  • 服务器返回一个特殊的HTTP响应，表示握手成功，状态码为101 Switching Protocols。

WebSocket通信：

  • 一旦握手成功，客户端和服务器之间建立了持久的双向通信通道。
  • 客户端可以使用WebSocket对象的send()方法发送消息给服务器。
  • 服务器可以使用WebSocket对象的send()方法发送消息给客户端。
  • 客户端和服务器都可以通过监听WebSocket对象的onmessage事件来接收对方发送的消息。

关闭WebSocket连接：

  • 客户端或服务器可以发送关闭帧来关闭WebSocket连接。
  • 客户端可以通过WebSocket对象的close()方法关闭连接。
  • 服务器可以通过发送关闭帧来关闭连接。
  • 使用WebSocket可以实现实时的双向通信，适用于聊天应用、实时推送、多人协同编辑等场景。同时，WebSocket还支持二进制数据的传输，可以用于传输文件、音频、视频等多媒体内容。

在使用WebSocket时，客户端和服务器之间的通信数据格式需要进行约定，以便双方正确解析和处理消息。常见的做法是使用JSON格式来传输数据，可以将数据序列化为JSON字符串发送，接收方再解析JSON字符串得到数据对象。此外，为了保证安全性，可以在WebSocket连接上使用TLS/SSL进行加密和认证。

6.Openfeigi基于什么通信协议封装的？

OpenFeign是一个基于Java的声明式HTTP客户端，它是对HTTP通信的封装，不限定具体的通信协议。

在底层，默认情况下，OpenFeign使用的是Java的URLConnection库来进行HTTP通信。URLConnection是Java标准库提供的用于建立和管理与远程服务器的连接的类。它支持多种协议，如HTTP、HTTPS、FTP等。

除了默认的URLConnection，OpenFeign还可以通过自定义配置来支持其他通信框架或库，如Apache HttpClient、OkHttp等。这些框架提供了更强大和灵活的功能，例如连接池管理、请求拦截器、表单编码等，可以根据需求选择适合的通信框架。

OpenFeign的优势在于其声明式的特性，通过注解和接口定义，将远程服务的调用细节隐藏在背后。开发者只需要定义接口，使用注解描述服务的URL、请求方法、请求参数等信息，OpenFeign会自动根据这些信息生成对应的HTTP请求。这样可以简化HTTP客户端的使用，并且提供了一致性、可维护性和可扩展性。

总结来说，OpenFeign并没有直接基于具体的通信协议进行封装，而是通过底层的HTTP库（默认是URLConnection）来实现对HTTP通信的封装。如果需要，可以通过自定义配置来切换到其他更强大的通信框架。

7.设计一个扫码登录的流程和逻辑？

  1. 生成登录标识：
    用户请求扫码登录，服务器生成一个唯一的登录标识（如UUID）。
    将该登录标识与当前用户的身份关联，并保存在服务器端。

  2. 生成二维码：
    服务器利用登录标识生成一个二维码图片，并返回给客户端。
    二维码中包含登录标识的信息。

  3. 用户扫码：
    用户使用扫码工具（如手机上的相机）扫描服务器返回的二维码图片。

  4. 检查登录状态：
    客户端通过轮询、长连接或WebSocket等方式向服务器查询当前登录状态。
    服务器检查登录标识是否有效，如果有效，则返回登录状态给客户端。

  5. 用户确认登录：
    当用户扫码后，客户端会不断查询登录状态，直到服务器返回已登录状态。
    服务器验证用户的身份，并将用户与登录标识关联起来。
    客户端提示用户登录成功，并跳转到相应的内容页面。
    需要注意的是，为了保证安全性和防止恶意登录，可以在扫码登录流程中加入以下措施：

二维码有效期限：可以设置二维码的有效期，过期后需要刷新二维码。
单次使用：每个二维码只能使用一次，防止重复扫描。
用户确认：用户扫码后，需要在客户端确认登录，避免未经用户许可的登录。
防止攻击：限制频繁扫码操作、限制同一IP地址的扫码次数等，以防止恶意攻击。

8.如何优化一条慢SQL？

优化慢SQL是提高数据库性能的重要步骤之一。下面是一些常见的优化方法：

  1. 使用索引：确保表中使用了适当的索引。通过分析查询语句和表结构，确定需要创建或修改哪些索引。索引可以加快查询速度，减少数据扫描。
  2. 优化查询语句：检查慢SQL语句，确保它们是最有效的。避免全表扫描、不必要的联接、子查询等操作。优化查询语句可以减少数据库的负载，提高查询性能。
  3. 分析执行计划：使用数据库提供的工具或命令，分析查询的执行计划。查看查询是否正确地使用了索引，是否存在性能瓶颈。通过分析执行计划，可以发现潜在的性能问题并进行优化。
  4. 缓存重复查询结果：对于一些频繁执行、结果不变的查询，可以考虑使用缓存来存储查询结果。这样可以减少数据库的访问次数，提高响应速度。
  5. 数据库参数调优：根据数据库的特点和系统负载情况，调整数据库的参数配置。例如，调整查询缓存大小、连接数、并发线程数等参数，以获得更好的性能。
  6. 优化数据模型：检查数据库中的数据模型，确保它们符合最佳实践和业务需求。合理设计表结构、关系连接、字段类型和大小等可以提高查询效率。
  7. 监控和调优工具：使用数据库提供的监控和调优工具，实时监测数据库性能并进行调整。例如，设置慢查询日志、监控数据库的响应时间和负载情况等。
  8. 分库分表：对于大型数据库或高负载的系统，可以考虑将数据进行分库分表。通过水平切分数据，可以提高查询性能和扩展数据库的处理能力。
  9. 硬件升级：如果数据库性能问题无法通过上述优化方法解决，可以考虑升级硬件设备，如增加内存、改进磁盘配置等。

综上所述，优化慢SQL需要综合考虑索引、查询语句、执行计划、缓存、数据库参数、数据模型、监控工具等多个方面。通过合理的优化策略和技术手段，可以显著提高慢SQL的执行效率和数据库的整体性能。

9.对高内聚低耦合的了解？

  • 高内聚（High Cohesion）指的是模块内部的组件或功能彼此紧密相关，共同完成特定的任务或目标。
  • 低耦合（Low Coupling）指的是模块与其他模块之间的依赖关系尽量降低。

高内聚低耦合是一种良好的软件设计原则，能够提高系统的可维护性、可测试性、可扩展性和可重用性

10.数据库三范式介绍？

第一范式（1NF）：确保数据库中的每个列都是原子性的，即每列中的数据不可再分。此外，1NF要求每个表都有一个主键来唯一标识每行数据。通过满足1NF，可以消除重复的数据项。

第二范式（2NF）：在满足1NF的基础上，保证非主键列与主键完全依赖，即非主键列必须完全依赖于主键，而不是依赖于主键的一部分。如果存在部分依赖，应该将非主键列拆分到独立的表中，并通过外键与原表建立关联。(消除部分依赖)

第三范式（3NF）：在满足2NF的基础上，确保非主键列之间没有传递依赖关系，即非主键列不能依赖于其他非主键列。如果存在传递依赖，应将其拆分到独立的表中，以保证数据的一致性。（消除传递依赖）

11.myBatis缓存机制

  • 二级缓存机制提升数据的检索效率，避免每一次数据检索都去查询数据库，
  • 一级缓存是SqlSession级别的缓存，也叫本地缓存，因为每一个用户在执行查询的时候都需要使用SqlSession来执行，为了避免每一次查询数据库，MyBatis把查询出来的数据缓存到SqlSession的本地缓存里面，后续的SQL如果在命中缓存的情况下，就可以直接从缓存读取这样的数据。

如果想要实现跨SqlSession级别的缓存，一级缓存是无法实现的，因此在MyBatis引入了二级缓存的一个设计，当多个用户在查询数据的时候，只要有任何一个SqlSession拿到了数据，就会放入到二级缓存里边，那么其他的SqlSession就可以从二级缓存里面去加载数据

实现原理：
一级缓存SqlSession对象里边有 Executor和Local Cache， 找executor找到了就返回数据，找不到就查找之后存入到Local Cahche里面,— 在分布式情况下可能会导致脏读问题？
二级缓存原理就是在原来Executor上去做了一个装饰（引入了CachingExector装饰器），流程就变成了：先查二级缓存，再查一级缓存，再查数据库。

MyBatis的二级缓存相对一级缓存来说，它实现了SqlSession之间的缓存数据共享，缓存粒度也能控制到namespace一个级别，还可以通过cache接口实现不同缓存的组合
